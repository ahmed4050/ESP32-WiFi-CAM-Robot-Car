📝 التوثيق الفني الشامل للمشروع (Technical Documentation)
1. نظرة عامة على المشروع (Project Overview)
نظام برمي متكامل يعتمد على متحكم ESP32-CAM للتحكم في سيارة روبوتية لاسلكياً عبر شبكة الـ WiFi. يدمج المشروع بين بث الفيديو المباشر من الكاميرا بزمن تأخير منخفض جدًا، والتحكم بالحركة عبر واجهة ويب تفاعلية تستخدم بروتوكول الـ WebSockets السريع، مع تفعيل ميزات متقدمة لضمان استقرار العتاد وحمايته من التعليق.

2. الميزات البرمجية الرئيسية (Key Features)
نظام العناوين الذكي (mDNS): يتيح الدخول للوحة التحكم عبر الرابط http://esp32cam.local دون الحاجة لمعرفة عنوان الـ IP الرقمي المتغير.

مؤقت الحراسة الذاتي (Watchdog Timer - WDT): نظام أمان يقوم بإعادة تشغيل الروبوت تلقائياً في غضون 10 ثوانٍ إذا تجمد الكود أو تعطل الاتصال.

إدارة التحكم المتعدد (Multi-Client Priority): يسمح بمتصلين متعددين لمشاهدة البث (حتى 4 مستخدمين)، ولكن يعطي صلاحية التحكم بالحركة لأول عميل متصل فقط (Master Controller) لحماية السيارة من تضارب الأوامر.

معايرة انحراف السيارة (Speed Trim Calibration): ميزة برمجية تتيح موازنة التفاوت الفيزيائي بين سرعة المحرك الأيمن والأيسر عبر متغير SPEED_TRIM لضمان السير في خط مستقيم.

البث الديناميكي الذكي: تحسين جودة بث الفيديو وعدد الإطارات (FPS) تلقائياً بناءً على قوة إشارة الـ WiFi المحيطة (RSSI).

3. المعمارية والمكتبات المستخدمة (Architecture & Dependencies)
يعتمد الكود على معمارية أحداث غير متزامنة (Asynchronous) لإدارة خادم الويب والـ WebSockets بالتوازي مع معالجة الكاميرا والمحركات:

المكتبة	الوظيفة الأساسية في الكود
<esp_camera.h>	تهيئة الكاميرا والتحكم في دقة الصورة والجودة والتقاط الإطارات.
<WiFiManager.h>	إنشاء نقطة اتصال (AP) ذكية عند أول تشغيل لضبط الـ WiFi دون تعديل الكود.
<ESPAsyncWebServer.h>	بناء خادم ويب غير متزامن عالي الأداء لا يعطل وظائف المتحكم الأساسية.
<AsyncTCP.h>	إدارة اتصالات الـ TCP غير المتزامنة وهي أساس عمل سيرفر الويب.
<ESPmDNS.h>	ربط عنوان الـ IP باسم النطاق المحلي الخاص بالروبوت.
<esp_task_wdt.h>	تفعيل الـ Watchdog لمراقبة استقرار المعالج وإعادة التشغيل الآمن.
4. المخطط المرجعي للدبابيس (Pinout Mapping)
أ) دبابيس الكاميرا الافتراضية (AI-Thinker Module)
تمت تهيئة الكاميرا بالدبابيس القياسية للوحة AI-Thinker لضمان استقرار نقل البيانات الموازية للبث الرقمي:

دبابيس البيانات: Y2 إلى Y9 مربوطة بالدبابيس العتادية (من GPIO 5 إلى GPIO 35).

دبابيس التحكم: XCLK (0)، PCLK (22)، VSYNC (25)، HREF (23).

التحكم بالطاقة والاتصال: PWDN (32)، RESET (-1)، SIOD (26)، SIOC (27).

ب) دبابيس المحركات والإضاءة (Actuators Pinout)
المحرك الأيمن / الأيسر: تم تخصيص الدبابيس البرمجية للتحكم باتجاهات المحركات وسرعتها عبر إشارات الـ PWM (تختلف حسب التوصيل الفيزيائي المعتمد في سيارتك).

الفلاش (Flash LED): مربوط على الدبوس القياسي GPIO 4 لتوفير إضاءة ليلية للكاميرا عبر واجهة الويب.

5. بروتوكول تبادل البيانات (Data Packet Protocol)
يتم تبادل البيانات بين واجهة الويب والمتحكم عبر رسائل نصية قصيرة بصيغة صيغ مخصصة أو JSON عبر الـ WebSocket (منفذ 82):

أ) الأوامر المستقبلة من الواجهة (Inbound Commands):
FORWARD, BACKWARD, LEFT, RIGHT, STOP: لتوجيه حركة السيارة.

SPEED:X: لتعديل السرعة الافتراضية (حيث X قيمة بين 0 و 255).

TRIM:X: لتعديل قيمة موازنة الانحراف برمجياً.

FLASH:ON / FLASH:OFF: للتحكم في كشاف الإضاءة.

ب) تقارير الحالة المرسلة للواجهة (Outbound Diagnostics):
يتم إرسال حزمة JSON دورية كل 5 ثوانٍ إلى المتصفحات المفتوحة لتحديث لوحة البيانات:

JSON
{
  "heap": 124560,    // حجم الذاكرة العشوائية الحرة بالبايت
  "rssi": -55,       // قوة إشارة الـ WiFi بالديسيبل
  "clients": 2       // عدد الأجهزة المتصلة حالياً بالروبوت
}
6. وظائف الأمان وإدارة الذاكرة (Memory & Safety Management)
دالة ws.cleanupClients(): تستدعى في الـ loop كل 5 ثوانٍ لتنظيف الذاكرة بشكل فوري من اتصالات الـ WebSockets الميتة أو المنقطعة، مما يمنع ثغرة تسريب الذاكرة الشهيرة في ESP32-CAM.

الـ Watchdog Reset: يتم تصفير العداد بانتظام عبر esp_task_wdt_reset(). في حال دخول الكود في حلقة مفرغة (Infinite Loop) أو تعليقه لأكثر من 10 ثوانٍ، يتدخل المعالج عتادياً ويقوم بعمل Hard Reset.

مراقبة الذاكرة الصارمة: يطبع الكود تحذيراً عبر الـ Serial Monitor إذا انخفضت الذاكرة الحرة (Free Heap) عن 50,000 بايت لتنبيه المطور قبل حدوث انهيار للنظام.

7. دليل التشغيل ومعالجة الأخطاء (Troubleshooting Guide)
المشكلة: السيارة لا تسير بخط مستقيم وتنحرف يميناً أو يساراً.

الحل: استخدم السلايدر الخاص بالـ TRIM في واجهة الويب أو عدل متغير SPEED_TRIM في الكود لتقليل سرعة المحرك الأسرع حتى تتوازن السيارة تماماً.

المشكلة: البث متقطع أو يتوقف فجأة عند ابتعاد الروبوت.

الحل: الكود يقلل الجودة تلقائياً لحماية البث، لكن تأكد من فحص قيمة الـ RSSI في السيريال؛ إذا كانت أقل من -80 dBm فهذا يعني أن الروبوت خارج نطاق التغطية الجيدة للـ WiFi.

المشكلة: لا يمكنني الدخول عبر esp32cam.local.

الحل: تأكد أن هاتفك أو كمبيوترك متصل بنفس الشبكة التي يتصل بها الروبوت. بعض أنظمة أندرويد القديمة قد تتطلب تنزيل تطبيق يدعم الـ mDNS أو استخدام الـ IP الرقمي المطبوع في الـ Serial Monitor كبديل احتياطي.
